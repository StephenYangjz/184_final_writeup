CS 184 Final Project: Snow Simulation & Interactions

By Christian Kim, Prince Wang, Stephen Yang, Rory Na

Abstract

Snow is beautiful but challenging to simulate. In this project, we simulate snow in real-time and create an efficient and relatively large-scale rendering of snow object interactions. By referencing past work on snow physics, we implemented snow simulation in Taichi, a platform that allows for parallelization and GPU acceleration. On top of it, we added extra features such as Gaussian and randomized terrain generation with newly added mountain/soil substance, wind simulation (both random and cursor controlled), and a user-friendly GUI. To illustrate our snow physics, we achieved relatively realistic snow and water dissolution simulation and gradual snowfall simulation in real-time thanks to the optimizations of the Taichi programming language.

Background

On one hand, snow simulation is useful as a visual and artistic project to render beautiful snow scenes. On the other hand, we want to see the limits of real-time rendering and create efficient and relatively large-scale renderings of snow object interactions. However, because snow possesses material characteristics of both a fluid and a solid, it can be very challenging to simulate. We proposed many ways to start the project, such as developing it from existing 184 projects, using Houdini, and using the Unity game engine. However, none of those approaches are ideal because of either efficiency issues, instability, or setup complexity. Among many technical approaches, we also decided to implement the Material Point Method based on a 2013 paper by Stomakhin et al. MPM offers efficient runtime and control over crucial properties of the material, because it aligns well with the content covered in this course.

We started off testing various particle simulation methods in C++. However, to accomplish what we wanted to simulate such as snowfall on mountains, using C++ could be very slow and hard to optimize and parallelize. So we explored many other options and decided to use the Taichi programming language, an extension to Python that enables GPU accelerated matrix manipulations and scene renderings.

Technical Approach - Snow Physics

We started with a simple Taichi simulation example with water, jelly, and snow materials. From there, we removed the original snow implementation and worked to completely reimplement our own MPM-based snow simulation physics according to the paper A material point method for snow simulation.

In our code, the sub_step() function is the main function that handles change in the states of particle and grid over a timestep △t, in which we derived and implemented all the above procedures from scratch. The sub_step() functions contain code that runs all ten steps of the algorithm listed above. Along with the main sub_step() function, we also implemented a handful of helper functions to handle specific calculations in the paper such as calculating ▽ψ and ▽w.

One of the main distinctions in our implementation is that rather than having class objects for things like the particle and grid, we encode all the information of the system such as the particle's, position, velocity, forces, and the corresponding grid components into globally scoped scalar, vector, and matrix fields. We believed that this approach would be more efficient and would provide the best cross-compatibility with the physics of water material that was already implemented in our starting code. However, using this approach was also much more complicated to work around as the bookkeeping for the particle and grid states is less straightforward.

Going step by step through the paper：

1) Rasterize particle data to the grid. We begin by rasterizing the particle's mass and velocity to the grid by scaling the particle's mass and velocity using a weighting function that performs different calculations depending on the particles' distances from the grid location, updating values of all grid locations that are nearby the particle.

2) Compute particle volumes and densities. We bypass the second step by setting the volume and density of all particles to be a constant value at the start of the program.

3) Compute grid forces. In the third step, we compute the grid forces using the equations outlined in the paper, which involves the partial derivative of the elastoplastic energy function with respect to elastic force as well as the gradient of the weighting functions.

4) Update velocities on the grid. The fourth step updates the grid velocities by using the grid forces calculated above by multiplying by dt and dividing by the grid mass.

5) Grid-based body collisions. We skip the fifth step in our implementation as we don't have any rigid structures in our simulation.

6) Solve the linear system. We "perform" step six, but since we use explicit integration, we don't actually perform any changes in this step.

7) Update deformation gradient. In step seven, the deformation gradient is updated, which involves setting the elastic and plastic forces that the particle is affected by. This is done by calculating the force with the assumption that all the force is elastic, taking the singular value decomposition of that elastic force, clamping the sigma values of that SVD to values that correspond to variables that define the snow behavior, and then reconstructing the elastic force and plastic force using this new sigma value.

8) Update particle velocities. We translate the grid velocities back to the particle velocities in step 8 by multiplying the grid velocity by the same weighting function as in step 1. The particle's velocity is decomposed to a PIC part and a FLIP part, where the PIC part is simply the grid velocity translated back to particle velocity, while the FLIP part is the change in the grid velocity from the previous time step's grid velocity translated to the particle's velocity. Then, the final particle velocity is calculated by weighting the PIC part and FLIP part depending on an alpha value that the paper sets as 0.95.

9) Particle-based body collisions. Step 9 is skipped for the same reason as for step 5.

10) Update particle positions. Finally, we update the particle positions in step 10 by adding the calculated velocity in step 8 with it and adding that to the previous particle position.

The biggest hurdle we ran into was getting incorrect-looking behavior after the implementation of the physics. Since there are just so many calculations going on under the hood in grid space where the values aren't easily interpretable and with no reference values to compare them to, debugging required scanning through the entire code and looking for any small issues. Furthermore, because Taichi is new to all of us in the group and there isn't great documentation or help online for it, there were points in our programming where unexpected behavior by Taichi ended up causing problems.

Ultimately, although our physics simulation is relatively stable and acts reasonably, it still appears to differ slightly from the kind of snow that was given in Taichi default implementations. In particular, we think that the distribution of elastic and plastic deformation is slightly stiffer. The elastic force appears to be a little stronger than the original implementation, resulting in "stretchier" snow. We've spent many hours figuring out why, but all of our calculations seem to be in order and our constants (like those for the critical stretch and critical compression, which should characterize the elastic and plastic deformation) match those defined in the paper. Thus, our speculation is that the paper inherently deals with snow physics in different ways than the previous implementation, which is also reasonable.

Technical Approach - Additional Features

Besides implementing snow physics, we also added numerous additional features. We created a non-moveable but interactable soil substance by providing a high hardening coefficient and skipping procedures related to position updates. To create realistic scenes, we used terrain generation methods to get an interactable mountain from the social substance. Specifically, we used Gaussian functions to simulate a mountainous region with two peaks. In addition, we also implemented a procedural random mountain generation function that uses a random function to connect the provided starting and ending position, with limited height and slope.

We added gradual random wind simulation and cursor-directed wind simulation by applying incremental horizontal acceleration, limited by the wind strength constant, adjusted on the GUI. The actual wind accelerations are updated procedurally with random functions to prevent unrealistic sudden wind direction changes.

We also implemented a gradual snowfall simulation on the mountain, similar to our pygame 2d implementation in the milestone. Since Taichi does not natively support adding new blocks while rendering, this is achieved by hanging a large piece of snow in the sky and simulating the right amount of wind to let them fall freely part by part.

Finally, using Taichi utility functions, we added a user-friendly GUI to allow for preset selection and change numerous physics parameters such as wind and gravity strength as well as enable/disable random wind.
